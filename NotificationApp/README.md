# NotificationApp

Учебный проект, который отвечает на вопрос: как правильно зарегистрировать в DI-контейнере несколько реализаций одного интерфейса - и что будет, если добавить фильтр перед отправкой?

## Что происходит при запуске

`NotificationService.NotifyAllAsync` получает сообщение, прогоняет его через `ProfanityFilter` и - если прошло - рассылает сразу по трём каналам: Email, SMS и Push. Каждый канал реализует `INotificationSender`, и сервис вообще не знает, сколько их и каких.

```text
=== Отправка обычного сообщения ===
[Email] Отправлено на user@example.com: Ваш заказ подтверждён
[SMS]   Отправлено на user@example.com: Ваш заказ подтверждён
[Push]  Отправлено на user@example.com: Ваш заказ подтверждён

=== Попытка отправить сообщение с запрещённым словом ===
[Фильтр] Сообщение заблокировано
```

## Про жизненные циклы

Интересная часть - не сами отправители, а то, как они зарегистрированы:

- `INotificationSender` (`EmailSender`, `SmsSender`, `PushSender`) - **Transient**: новый экземпляр на каждый запрос, потому что у отправителей нет состояния, которое нужно переиспользовать
- `ProfanityFilter` - **Singleton**: создаётся один раз, потому что хранит только read-only набор запрещённых слов - никакого мутируемого состояния нет
- `NotificationService` - **Scoped**: один экземпляр на логический scope (в веб-приложении это был бы HTTP-запрос)

Три сендера регистрируются каждый отдельным вызовом `AddTransient<INotificationSender, ...>`, и DI-контейнер сам собирает их в `IEnumerable<INotificationSender>`, когда `NotificationService` просит этот тип в конструкторе.

## Запуск

```bash
dotnet run --project NotificationApp
```
